# 分专业 AI 风险数据库说明

更新时间：2026-05-09

这个文件是 `major_risk_database.json` 的阅读版。它的目的不是替代网页，而是作为后续深化网页的“数据底稿”：每个专业类都要能追溯到风险机制、抗风险机制、证据锚点、可反驳点和置信度。

## 数据库字段

| 字段 | 含义 |
|---|---|
| `mainlandCategory` | 中国大陆本科专业类/常见专业组合 |
| `overseasEquivalent` | 美英常见对应，用于留学和海外求职理解 |
| `overallRisk` | 总体入口风险，不等于行业消失概率 |
| `confidence` | 当前判断置信度，高置信度通常意味着有较明确的任务暴露数据或机制证据 |
| `observedExposureAnchors` | Anthropic observed exposure 中可对应的职业数据 |
| `riskMechanisms` | 为什么该方向入门岗位会被 AI 压缩 |
| `resilienceMechanisms` | 为什么该方向不容易被直接替代 |
| `decisionCells` | 网页主矩阵可直接使用的短结论 |
| `higherRiskEntryTasks` | 本科生/硕士生入门时较危险的任务 |
| `saferSubdirections` | 更稳或更值得转向的子方向 |
| `portfolioEvidence` | 学生应拿什么作品或经历证明自己不只是做标准任务 |
| `counterarguments` | 反向观点，用于防止结论过度武断 |
| `weaknessesInCurrentAssessment` | 当前数据库最需要补证据的地方 |

## 已细化样板：计算机/软件/数据/人工智能

计算机方向已新增 `subfields` + `entryRoles` 样板，并经过 Claude 独立审核。审查后新增“游戏开发/实时图形/交互系统”，并修正前端、测试、云/SRE 等评级。当前包含：

| 子方向 | 风险层 | 典型入口岗位 | 一句话判断 |
|---|---|---|---|
| 应用后端 / 全栈业务开发 | 高 | Java/Go 后端、全栈、企业应用开发 | 普通业务代码仍有需求，但低复杂度 CRUD、接口粘合和页面配套会最先被压缩。 |
| 前端 / 移动端应用开发 | 中高 | Web 前端、移动端、小程序、UI 工程化 | 页面实现和组件拼装高危，但复杂状态、性能、跨端和产品体验仍有门槛。 |
| 游戏开发 / 实时图形 / 交互系统 | 中高 | 游戏客户端、游戏服务器、引擎、TA、图形渲染 | 游戏逻辑和工具脚本会被 AI 加速，但引擎、图形、性能和网络同步仍有壁垒。 |
| 软件测试 / QA / 测试开发 | 中高 | 软件测试、测试开发、QA、自动化测试 | 手工测试接近高风险，测试开发和质量工程取决于平台化与发布风险控制。 |
| 数据分析 / BI / 低代码报表 | 高 | 数据分析师、BI 工程师、商业分析、报表开发 | 工具型 BI 报表高度暴露；能做实验、因果和指标治理的分析岗位才更稳。 |
| 数据工程 / 数据平台 / 数据库应用 | 中高 | 数据开发、数仓、数据平台、数据库应用 | ETL 脚本会被增强，但数据质量、平台稳定性和成本治理仍形成门槛。 |
| AI / 大模型应用工程 | 中高 | AI 应用、大模型应用、RAG、Agent 应用 | 机会很热，但只做模型调用、RAG 拼装和演示项目会迅速同质化。 |

| 云计算 / SRE / DevOps / 平台工程 | 中高 | 云计算、DevOps、SRE、平台工程 | 脚本和配置会被自动化，但线上可靠性、成本、容量和事故责任仍是门槛。 |
| 系统软件 / 基础软件 / 编译器 / 数据库内核 | 中 | 基础软件、数据库内核、编译器、OS/存储 | 代码生成会提效，但深层系统约束、性能和正确性让基础软件更稳也更难。 |
| 算法 / 机器学习 / 模型训练 / AI 基础设施 | 机会 | 算法、机器学习、大模型训练、AI Infra、研究工程 | 高机会但高门槛；普通调参和论文复现风险高，稀缺的是数据、评测、训练和系统能力。 |

## 已细化样板：自动化类/机器人工程/智能制造

自动化/机器人/智能制造方向已新增 `subfields` + `entryRoles` 样板，并经过 Claude 独立审核。外部证据锚点包括 IFR World Robotics 2025 对工业机器人装机和中国市场占比的统计、工信部等 17 部门《“机器人+”应用行动实施方案》对机器人应用场景的政策导向，以及 BLS 对机电一体化技术人员和工业工程师的职业背景。Claude 审核认为原先容易低估 AMR/AGV、PLC 基础集成和重复测试验收的入门压缩，也指出控制算法不能因为仿真易自动化就整体上调。已据此将控制算法降为中、AMR/AGV 升为高，并补入工业机器人示教/离线编程和自动化系统测试/FAT/SAT 报告。当前分布：4 高、5 中高、7 中、3 低、3 机会。

| 族群 | 子方向 | 风险层 | 一句话判断 |
|---|---|---|---|
| 高暴露 | 机器人算法 / ROS / 仿真 Demo | 高 | 开源栈和论文复现会迅速同质化。 |
| 高暴露 | 数字孪生 / 仿真大屏 / 低代码自动化 | 高 | 看板、报表和演示系统最容易模板化。 |
| 高暴露 | AMR / AGV / 仓储物流自动化 | 高 | 导航 Demo 拥挤，入门价值要看场地落地。 |
| 高暴露 | 自动化系统测试执行 / FAT / SAT 报告 | 高 | 重复测试、记录和报告高度流程化。 |

## 后续深化网页的建议

1. 网页主矩阵继续只展示 `decisionCells`，保持轻量。
2. 每个专业类可以做一个“展开详情”或侧边面板，读取：
   - `riskMechanisms`
   - `resilienceMechanisms`
   - `higherRiskEntryTasks`
   - `saferSubdirections`
   - `portfolioEvidence`
3. 增加“证据强弱”视觉标记：高置信度、中置信度、中低置信度。
4. 对低置信度专业类优先补证据，尤其是新工科、机器人、生物医学和材料化工。
5. 后续可以把 Anthropic observed exposure 数据单独做成 `occupational_exposure.csv`，再由网页引用。
6. madeye/jobs 与 karpathy/jobs 只作为职业入口交互方式参考，不采用其中 AI 估算职业数据作为本数据库证据。

## 当前需要重点复核的地方

- 大陆就业市场与美国 O*NET/BLS/Anthropic 数据之间存在映射误差。
- observed exposure 来自 Claude 使用数据，可能低估 EDA、CAD/CAE、实验自动化、工业软件等专用工具。
- 高考、考研、读博的时间窗口不是预测年份，而是“风险观察窗口”。
- “低风险”不等于高薪或好就业，只表示直接被 AI 替代的速度较慢。
- “机会”不等于容易，通常意味着高波动、高门槛、高分化。
